The integration problem

Why MCP exists

Without a shared protocol, every AI application needs custom glue code for every data source and every external action. MCP gives AI applications a common way to discover tools, read resources, and use reusable prompts.

🔌

Before MCP

Every assistant needs one-off integrations: GitHub adapter here, database adapter there, filesystem connector somewhere else.

🧭

With MCP

A host connects to one or more MCP servers. The host discovers capabilities, then routes requests through a standard protocol.

🛡️

The catch

MCP makes tool access easier, so permissions, approvals, scoping, logging, and prompt-injection defenses become more important.

Mental model

MCP architecture in one picture

Think of MCP as a client-server protocol around an AI host. The host manages the model experience. MCP servers expose capabilities. The client connection handles discovery, calls, and results.

1
User asksThe human asks for an action or answer: “Create a GitHub issue” or “Summarize this schema.”
2
Host decidesThe AI app decides whether it needs a tool, resource, or prompt from an MCP server.
3
Client routesThe MCP client sends JSON-RPC requests to the relevant MCP server.
4
Server respondsThe server returns structured content, data, or action results.
5
LLM continuesThe host feeds the result back into the conversation so the model can answer or continue.

Lifecycle

What actually happens during an MCP session

The protocol flow is simple but important. Initialization negotiates capabilities. Discovery tells the host what exists. Tool calls execute structured operations. Notifications keep the host updated.

🤝

Initialize

Client and server exchange protocol versions, identities, and supported capabilities.

🔎

List

The client calls list methods like tools/list, resources/list, or prompts/list to discover what the server exposes.

⚙️

Call or read

The client invokes tools/call or reads resources after the host chooses what it needs.

🔔

Notify

Servers can notify clients when available tools or other capabilities change.

Core concepts

Tools, resources, and prompts

Most confusion around MCP disappears when these three server-side primitives are separated clearly.

PrimitiveWhat it isExampleWhen to use it
ToolAn executable action with an input schema.create_issue, query_database, send_slack_messageWhen the AI needs to do something.
ResourceReadable context or data exposed by a server.file contents, DB schema, logs, docsWhen the AI needs to read information.
PromptA reusable template or workflow instruction.debug_api_error, review_pr_templateWhen you want a repeatable domain-specific workflow.
Guardrails

Security is not optional

MCP gives models access to real tools and real data. That is powerful, but it means bad tool descriptions, broad permissions, or prompt injection can become real operational risk.

🔐

Scope every tool

A tool named run_sql is dangerous. A tool named read_customer_by_id with a strict schema is easier to reason about.

👁️

Require approval for writes

Read-only tools can often be safer. Destructive or external write actions should require explicit user confirmation.

🧼

Treat tool output as untrusted

Documents, web pages, tickets, and issue bodies can contain prompt injection. Do not blindly follow instructions found in tool output.

Production readiness checklist

✓ Clear tool names and descriptions
✓ Narrow input schemas with validation
✓ User approval for writes and destructive actions
✓ Secrets never returned to the model
✓ Per-user authorization and least privilege
✓ Rate limits, timeouts, and retries
✓ Audit logs for tool calls
✓ Observability for failures and latency

Mistakes

MCP anti-patterns

One giant tool

“execute_anything” forces the model to reason about too much freedom. Split broad tools into smaller scoped actions.

Vague schemas

If parameters are loose strings, the model can produce bad calls. Use enums, constraints, clear descriptions, and validation.

Too many tools

Dumping dozens of similar tools into context makes selection harder. Prefer a small, curated tool surface.

Returning secrets

Never return tokens, credentials, private keys, or internal secrets as tool output.

No source boundaries

Resource servers should know which roots, folders, projects, users, or tenants are allowed.

No audit trail

You need to know who asked, what tool was called, with which parameters, and what happened.

Q&A

MCP FAQ

Is MCP the same as an API?
Not exactly. MCP can wrap APIs, files, databases, and workflows, but it also standardizes discovery, schemas, resources, prompts, and the client-server conversation used by AI hosts.
Is MCP the same as function calling?
Function calling is usually a model feature. MCP is a protocol around how an AI application discovers and invokes tools from external servers.
When should I expose a tool?
Expose a tool when the assistant needs to perform an action or compute a result. If it only needs to read context, a resource may be a better fit.
Should write actions be automatic?
Usually no. Write actions that affect external systems should be clearly scoped and often require user approval.
Can MCP connect to private data?
Yes, but the server must enforce authentication, authorization, tenant boundaries, and safe output handling.
Does MCP remove security risk?
No. It standardizes integration, but security still depends on scopes, validation, approvals, auditing, and how the host handles untrusted content.
What should a first MCP server do?
Start small: one read-only resource or one safe tool. For example, “list recent project notes” or “search internal docs.”
What is the difference between STDIO and HTTP?
STDIO is common for local servers launched by a desktop or IDE host. Streamable HTTP is used for remote servers that can serve multiple clients.
Glossary

Core MCP terms

MCP Host

The AI application that manages the user experience and model context.

MCP Client

The component inside the host that maintains a connection to one MCP server.

MCP Server

A program that exposes tools, resources, and prompts to MCP clients.

Tool

An executable function with a schema, such as query_database or create_issue.

Resource

Readable contextual data, like file contents, logs, docs, or schemas.

Prompt

A reusable template that guides a repeated interaction or workflow.

JSON-RPC

The message format used for requests, responses, and notifications.

Transport

The communication channel, commonly STDIO for local servers or HTTP for remote servers.

Capability negotiation

The initialization step where client and server declare what they support.
